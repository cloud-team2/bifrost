Q. 로그만 주면 진단되나?

로그를 그냥 LLM에 던지지 않는다 — Evidence-grounded RCA

단순 로그 요약은 “그럴듯한 환각”을 만든다. Bifrost는 증거 수집 → 카탈로그 매칭 → 증거 게이트 → 신뢰도 산정의 결정론적 파이프라인을 거친다.

✕Log dump → LLM (안티패턴)

원인을 자유 생성 → 검증 불가, 재현 불가, 같은 입력에 다른 답. RCA가 아니라 “요약”.

→ 신뢰 불가
✓Evidence-grounded 파이프라인

증거 정규화 → 35종 root-cause 카탈로그와 대조 → evidence matrix(required/supporting/negative) 충족 점수화 → 신뢰도 산정.

→ 근거·재현 가능
증거 부족 시 단정 대신 abstention(기권) — 실측 35건 중 5건 보류, 카탈로그-외 원인(환각) 0건
벤치마킹: ITIL Problem Management의 evidence 기반 진단 + RAG(Lewis et al., NeurIPS 2020)의 grounding 원리를 RCA 파이프라인에 적용

환각 통제 · RCA 국제 표준

RCA 방식은 글로벌 IT 운영 표준을 벤치마킹했다

“정해진 원인 목록에서 증거로 고른다”는 우리 임의 설계가 아니라, 수십 년 검증된 ITSM·품질관리 표준을 LLM RCA에 이식한 것이다.

글로벌 레퍼런스업계는 이렇게 한다 (원리)→ Bifrost 적용
ITIL 4 Problem Mgmt+ KEDB · Axelos근본원인 분석 후 Known Error로 등록해 재발 시 재사용35종 root-cause 카탈로그 + KEDB(증상·검증된 조치·롤백 절차)
5 WhysToyota Production System표면 증상이 아니라 더 깊은 근본원인까지 파고든다증상(Connector FAILED)을 입증된 심층 원인(DB timeout 등)으로 강등
Evidence-groundingRAG · Lewis et al. 2020근거 문서에 연결되지 않은 생성은 신뢰하지 않는다evidence matrix로 required 증거 미충족 시 신뢰도 상한(cap)
ISO/IEC 20000-1IT 서비스 관리incident → problem → known error의 표준 프로세스Classifier→RCA→Verifier→Report로 ITSM 흐름 자동화
핵심 원칙 — RCA Assistant, not Engine: LLM은 원인을 ‘생성’하지 않고, 카탈로그에서 증거로 ‘선택·설명’한다
출처: ITIL 4 Problem Management(Axelos 2020) · 5 Whys(Ohno 1988) · RAG(arXiv:2005.11401) · ISO/IEC 20000-1:2018

Q. Rollback / 실패 시나리오
