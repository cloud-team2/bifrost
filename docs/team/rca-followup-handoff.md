## 이슈별 쉬운 설명 (비전공자용)

각 이슈가 정확히 무슨 작업인지 1~2문장으로 풀어 썼다. (각 GitHub 이슈 맨 위에도 동일하게 적혀 있다.)

1. **#882 실행 깊이 제어** — 지금은 "상태 좀 봐줘" 같은 쉬운 질문에도 AI가 매번 모든 분석 절차를 끝까지 돌려서 느리고 비용이 큽니다. 질문이 쉬우면 짧게, 어려우면 깊게, 질문 난이도에 맞춰 AI가 하는 일의 양을 자동으로 조절하도록 만드는 작업입니다.
2. **#883 실행 관측성** — AI가 한 번 분석할 때 어느 단계에서 시간·비용이 얼마나 들었고 도구를 몇 번 썼는지 기록이 남지 않습니다. 나중에 "왜 느렸지/비쌌지"를 추적할 수 있도록 실행 영수증(기록)을 남기는 작업입니다.
3. **#884 threshold 레지스트리** — "이 수치를 넘으면 경고" 같은 기준 숫자들이 코드 곳곳에 흩어져 박혀 있어, 누가 왜 그 값으로 정했는지 알 수 없습니다. 이 기준값들을 한 곳에 모아 값·이유·담당자·변경이력과 함께 관리하는 작업입니다.
4. **#885 재현성 스키마** — 지금은 "AI가 그때 어떤 모델·설정으로 판단했는지" 기록이 부족해, 같은 장애를 나중에 다시 분석하면 결과가 달라질 수 있습니다. 당시 판단을 똑같이 재현할 수 있도록 사용한 모델·버전·설정을 기록에 고정하는 작업입니다.
5. **#886 자동 롤백** — AI가 자동으로 조치를 했다가 실패하면, 지금은 사람이 직접 손으로 되돌려야 합니다. 조치가 실패하면 시스템이 스스로 원래 상태로 되돌리는(undo) 안전장치를 만드는 작업입니다.
6. **#887 gold set·라벨링** — AI의 장애 원인 추측이 실제로 맞는지 채점하려면 "정답지"가 필요한데 지금은 없습니다. 실제로 해결된 과거 장애들에 대해 사람이 "진짜 원인은 이것이었다"라고 정리한 정답 모음을 만드는 작업입니다.
7. **#888 AC@k 평가** — AI는 장애 원인 후보를 여러 개 순위로 내놓는데, 그 순위 안에 진짜 원인이 얼마나 잘 들어오는지(예: 상위 3개 안에 정답이 있는 비율)를 점수로 측정하는 작업입니다. (6번 "정답지"가 있어야 가능)
8. **#889 ECE 캘리브레이션** — AI가 "이 원인일 확률 80%"라고 말할 때 실제로도 그만큼 맞는지 검증하는 작업입니다. 과하게 자신만만한 구간을 찾아 AI의 신뢰도 표현을 현실에 맞게 보정합니다. (6·7번 필요)
9. **#890 drift 감시** — 시간이 지나면서 AI의 판단 경향이 슬슬 이상해지거나(특정 원인만 자꾸 지목한다든지), 사람이 AI 결과를 고치는 일이 늘어나는 것을 자동으로 감지해 알려주는 작업입니다. (6·8번 필요)
10. **#891 SLI 정의** — 지금 알림은 "lag 5000 초과" 같은 내부 기술 수치를 기준으로 울립니다. 그 대신 "사용자 입장에서 데이터가 제때, 빠짐없이 도착했는가" 같은 실제 체감 품질 지표를 정의하는 작업입니다.
11. **#892 SLO burn-rate 알림** — 사용자 체감 품질이 빠르게 나빠질 때만 긴급 호출(page)을 보내고, 단순 기술 수치 초과는 덜 급한 알림으로 내려서 "불필요한 알림 폭탄"을 줄이는 작업입니다. (10번 필요)
12. **#893 인과/상관 증거 태그** — "같이 일어났다"와 "이것이 원인이다"는 다릅니다. AI가 단순히 동시에 나타난 현상을 진짜 원인으로 착각하지 않도록, 시간 순서(원인이 먼저 일어났는지)를 따져 증거의 무게를 다르게 매기는 작업입니다.
13. **#894 KEDB 확장** — 장애 원인 목록이 지금은 고정된 사전에 불과합니다. 원인마다 "담당자·증상·검증된 해결법·재발 횟수·관련 장애"를 계속 쌓아가는, 살아있는 운영 지식 DB로 키우는 작업입니다.

## 작업 방식 (팀 컨벤션)

각 이슈 본문에 **현재 상태(develop) + 파일 경로 + 해야 할 일 + 완료 조건**이 들어 있어 이슈만 보고 작업할 수 있다. 더 깊은 근거가 필요하면 이슈가 가리키는 [`rca-standards-review.md`](../design/rca-standards-review.md) 해당 §를 참고한다.

[`docs/team/git-convention.md`](./git-convention.md) 기준:

```
1. 이슈 확인 (#882 등)
2. develop에서 브랜치 생성:  feat/#882
3. 커밋:  #882 [feat] <메시지>
4. PR base = develop,  제목 = #882 [feat] <메시지>
5. Squash and merge
```

- 모든 이슈는 `feat` 라벨, develop이 base.
- ai-service 이슈와 Spring 이슈는 영역이 분리돼 있어 병렬 진행 가능.
- 한 단계가 여러 PR로 쪼개져도 무방 — 완료 조건 체크박스를 기준으로 한다.
